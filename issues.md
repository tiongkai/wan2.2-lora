# Engineering Issues

This backlog tracks the engineering work needed to turn the current Wan2.2
synthetic-video experiment harness into a repeatable data-generation pipeline.

## Resolution Summary

Status: original Issues 1-9 are implemented on branch
`engineering-pipeline-hardening`. Code review found follow-up issues CR-1
through CR-5 below that should be resolved before treating the branch as
production-ready.

- Issues 1-3 are resolved by the canonical manifest layer in
  `scripts/manifest.py`, manifest-producing generation scripts, and
  manifest-driven export in `scripts/annotate.py`.
- Issue 4 is resolved by workflow validation in `scripts/workflow_utils.py` and
  expected node-class checks in `scripts/generate.py` and
  `scripts/generate_i2v.py`.
- Issue 5 is resolved by Wan-aware frame validation and preprocess reports in
  `scripts/preprocess.py`.
- Issue 6 is resolved by manifest-based QA in `scripts/validate_dataset.py`,
  including blur, static, duplicate, black-frame, duration, frame-count, and
  resolution reporting.
- Issue 7 is resolved by long-form manifest support in `scripts/extend_video.py`,
  direct registration via `scripts/register_generated.py`, and whole/windowed
  export modes in `scripts/annotate.py`.
- Issue 8 is resolved by centralized category metadata in
  `scripts/categories.py`, including `self_injury`.
- Issue 9 is resolved by `configs/pipeline.yaml`, `scripts/pipeline_config.py`,
  `scripts/smoke_check.py`, and `PIPELINE.md`.

## Code Review Follow-Up Issues

These were found during subagent code review after the first implementation pass.

## CR-1: Non-Dry Export Can Crash On Empty Or Unknown Seeds

`scripts/register_generated.py` allows registered/manual outputs to have an empty
seed, but `scripts/annotate.py` casts every seed with `int(row["seed"])` while
building COCO JSON. A dry run can pass, `annotations.csv` can be written, and then
`annotations_coco.json` can fail with a seed conversion error.

### Completion Criteria

- Non-dry export succeeds for registered SVI/manual outputs with empty, missing,
  or non-numeric seeds.
- COCO export preserves seed information without forcing invalid integer casts.
- `annotations.csv` and `annotations_coco.json` are written atomically enough that
  partial export failure does not leave misleading output.
- Tests cover non-dry export for a registered output with an empty seed.

### Implementation Plan

1. Add a seed normalization helper in `scripts/annotate.py` that returns an int
   only when the value is safely numeric, otherwise returns an empty string or
   `None`.
2. Use the helper in `build_coco_json` instead of `int(row["seed"])`.
3. Consider making `register_generated.py --seed` optional but explicit in the
   manifest as unknown/null rather than an empty string.
4. Add a test that registers or constructs a manifest row with an empty seed and
   runs full `export_dataset(..., dry_run=False)`.

## CR-2: Long-Form Chaining Can Attach The Wrong Segment Under Concurrency

`scripts/extend_video.py` discovers each generated segment by scanning
`generated/clips/**/*.mp4` and selecting the newest file since the segment started.
If another ComfyUI job finishes during the same window, the long-form chain can
stitch and record the wrong video.

### Completion Criteria

- `extend_video.py` resolves segment files from the specific ComfyUI prompt
  history entry, not global newest-file discovery.
- Concurrent ComfyUI output cannot be accidentally selected as a chain segment.
- Segment paths recorded in the final manifest correspond to the prompt IDs
  generated by that `extend_video.py` run.
- Tests cover segment resolution from ComfyUI history and reject unrelated newer
  files.

### Implementation Plan

1. Change `gi.wait_for_completion` usage in `extend_video.py` to keep the returned
   history entry.
2. Use `scripts.manifest.comfy_history_output_paths` to extract the exact mp4 path
   for each prompt.
3. Keep newest-file fallback only as an explicit last resort with validation
   against the unique prefix for that segment.
4. Add a unique run identifier to segment prefixes, for example
   `long_chain/<run_id>/seg_XX`.
5. Add tests for history-based segment path extraction and for ignoring unrelated
   newer mp4 files.

## CR-3: Windowed Long-Form Export Does Not Produce Complete Window Semantics

`scripts/annotate.py --long-mode windows` creates multiple metadata rows from one
long video, but each row still points to the same full video. COCO export also
drops `source_video`, `window_start_frame`, and `window_end_frame`, so consumers
that only read COCO cannot recover the intended temporal window.

### Completion Criteria

- Windowed export either materializes real clipped window mp4 files or records
  temporal offsets in every export format.
- CSV and COCO exports both include source video and window start/end metadata
  when long clips are exported as windows.
- Window rows are unambiguous for fixed-length detector training consumers.
- Tests cover both CSV and COCO output for a long-form window export.

### Implementation Plan

1. Decide whether window export should be metadata-only or should physically
   create window clips.
2. If metadata-only, add `source_video`, `window_index`, `window_start_frame`, and
   `window_end_frame` to COCO `videos` or annotations.
3. If materializing clips, add an ffmpeg split step and make `filename` point to
   the generated window file.
4. Preserve the original long video path in `source_video`.
5. Extend tests to assert that CSV and COCO carry equivalent window information.

## CR-4: Export Allows Stale Or Unvalidated Clips Through QA Filtering

When `validation_report.csv` exists, `scripts/annotate.py` merges QA rows but
treats missing `qa_pass` values as passing. If new clips are generated after the
last validation run, those clips can be exported without QA.

### Completion Criteria

- When a QA report exists, export only includes rows with explicit `qa_pass ==
  true` by default.
- Unvalidated clips fail export or are excluded with a clear message.
- There is an explicit opt-in escape hatch, such as `--allow-unvalidated`, for
  exploratory exports.
- Tests cover stale QA reports where some manifest rows have no validation row.

### Implementation Plan

1. Change `_valid_export_rows` in `scripts/annotate.py` so missing `qa_pass` is
   not treated as passing when a QA report is present.
2. Add an `allow_unvalidated` parameter to `export_dataset` and a
   `--allow-unvalidated` CLI flag.
3. Report how many rows were excluded for missing QA.
4. Add tests for validated pass, validated fail, missing QA with default behavior,
   and missing QA with the opt-in flag.

## CR-5: QA Report Columns Are Not Stable For Failure Cases

`scripts/validate_dataset.py` returns early for black clips and exception cases
without all promised QA fields. If all rows fail in one path, the resulting CSV
can omit columns such as `blur_score`, `static_score`, `perceptual_hash`,
duration, resolution, or frame count.

### Completion Criteria

- `validation_report.csv` always includes the same QA columns, regardless of
  success, black frames, missing files, or exceptions.
- Failure rows use empty/null values for unavailable metrics instead of omitting
  columns.
- `manifest_validated.csv` can always join on stable QA columns.
- Tests cover all-failure reports for black videos, missing files, and exception
  cases.

### Implementation Plan

1. Define a stable QA result schema in `scripts/validate_dataset.py`.
2. Create a helper that builds result rows with default empty/null metric values.
3. Use that helper for success, black-frame, too-short, no-stream, missing-file,
   and exception cases.
4. Ensure duplicate detection works when `perceptual_hash` is missing or empty.
5. Add tests that validate report columns for each failure class.

## Issue 1: Generated Outputs Are Not Consistently Exportable

Current I2V/base/SVI outputs are written to folders such as
`generated/clips/fighting_i2v_base/`, while the exporter only scans
`generated/clips/<category>/<category>_manifest.csv`. Useful generated clips can
therefore be skipped during dataset export.

### Completion Criteria

- Dataset export discovers every supported generation run type: `t2v`,
  `i2v_base`, `i2v_lora`, `extend_chain`, and `svi`.
- Export no longer depends on one hardcoded folder shape.
- Exported dataset counts match the number of valid manifest rows on disk.
- A dry-run command can list all exportable clips before writing annotations.

### Implementation Plan

1. Define a common manifest discovery function that scans known manifest
   locations under `generated/`.
2. Update `annotate.py` to consume discovered manifests instead of
   `generated/clips/<category>/<category>_manifest.csv` only.
3. Add method filtering flags, for example `--methods i2v_base,svi`.
4. Add tests covering legacy T2V manifests and current I2V/base manifests.

## Issue 2: Manifest Filenames Do Not Match Actual Generated Files

The annotation exporter invents filenames such as
`synthetic_fighting_00000.mp4`, while generation writes files with ComfyUI
prefixes such as `fighting_i2v_base/sample_0000_*.mp4`. This can produce labels
that point to nonexistent files.

### Completion Criteria

- Every exported annotation references a real video file verified on disk.
- Manifest rows include the actual generated output path.
- Export fails clearly if a manifest row has `status=ok` but no matching file.
- Annotation code no longer constructs output filenames from category/id alone.

### Implementation Plan

1. Add an `output_path` field to all generation manifests.
2. Resolve saved ComfyUI outputs from `/history/{prompt_id}` wherever possible.
3. Fall back to prefix-based discovery only with explicit validation.
4. Update `build_annotation_csv` and `build_coco_json` to use `output_path`.
5. Add tests for missing files, valid files, and legacy manifests without
   `output_path`.

## Issue 3: No Canonical Run Manifest Schema

T2V, I2V, base, LoRA, long-chain, and SVI runs currently carry different implicit
metadata. This makes reproducibility, QA, and export brittle.

### Completion Criteria

- A canonical manifest schema exists and is used by all generation scripts.
- Each generation attempt records:
  - actual output path
  - category and class id
  - generation method
  - prompt and negative prompt
  - seed
  - start frame path, if applicable
  - frames, fps, width, and height
  - workflow template path and hash
  - model stack and LoRA metadata
  - status, error reason, prompt id, and created timestamp
- Existing manifests can be migrated or read through a compatibility layer.

### Implementation Plan

1. Add `scripts/manifest.py` with schema constants, validation helpers, and row
   builders.
2. Centralize category/class metadata in the same module or a config file.
3. Update `generate.py`, `generate_i2v.py`, and `extend_video.py` to write schema
   compliant rows.
4. Add a lightweight manifest validation command.
5. Add unit tests for required fields, default values, and legacy row coercion.

## Issue 4: ComfyUI Workflow Patching Is Fragile

Generation scripts patch hardcoded node IDs. A workflow re-export can silently
move or alter nodes, causing wrong fields to be patched or runs to fail late.

### Completion Criteria

- Workflow patching validates node IDs, expected fields, and expected node class
  names before queuing.
- Invalid workflow templates fail fast with a useful error.
- Current workflow JSONs pass validation in tests.
- Re-exporting a workflow requires updating one explicit patch spec, not
  scattered script code.

### Implementation Plan

1. Add a workflow validation helper that checks node existence, field paths, and
   optional class/type expectations.
2. Move patch specs into structured constants or config.
3. Call validation from `build_workflow` before applying patches.
4. Add tests for current T2V/I2V/base workflow templates.
5. Add negative tests for missing nodes and missing fields.

## Issue 5: Preprocessing Does Not Enforce Wan Training Constraints

Preprocessing trims clips to 5 seconds at 24 fps, but Wan training depends on
effective 16 fps frame availability and valid `4n+1` frame counts. The docs know
that 77 frames is the max valid length for 5-second clips, but the preprocessing
code does not enforce or report this.

### Completion Criteria

- Preprocessing reports effective frames at Wan's 16 fps rate.
- Requested `target_frames` are checked against actual clip duration.
- Invalid targets, such as 81 frames from a 5.0 second clip, fail before caching.
- Preprocessing supports explicit `--target-frames`, for example `33` or `77`.
- Output metadata records original duration, processed duration, fps, and
  effective Wan frames.

### Implementation Plan

1. Add Wan frame math helpers: effective frames at 16 fps and nearest valid
   `4n+1` length.
2. Extend `preprocess.py` with `--target-frames` and `--wan-fps`.
3. Validate each accepted clip against the requested target before writing it.
4. Emit a preprocessing report CSV per category.
5. Add tests for 33f, 77f, and failing 81f-on-5s cases.

## Issue 6: Generation Quality Validation Is Too Shallow

Current validation checks basic duration and black frames. It does not flag blur,
static clips, duplicates, scene drift, or whether clips should go to manual
review.

### Completion Criteria

- Validation reads canonical manifests and validates actual `output_path` files.
- QA report includes duration, resolution, frame count, black-frame status, static
  score, blur score, duplicate warning, and pass/fail status.
- Failed clips include a clear reason.
- Manifest rows can be updated or joined with QA status.
- Export can filter to `qa_pass=true`.

### Implementation Plan

1. Refactor `validate_dataset.py` to operate on manifest rows.
2. Add blur scoring using sampled grayscale variance or Laplacian variance.
3. Add static detection using sampled frame differences.
4. Add duplicate detection with sampled-frame perceptual hashes or compact image
   hashes.
5. Write `validation_report.csv` and optionally `manifest_validated.csv`.
6. Add tests using synthetic normal, black, static, blurry, and duplicate videos.

## Issue 7: Long-Form Outputs Are Outside The Dataset Lifecycle

`extend_video.py` and SVI workflows produce useful long-form videos, but their
outputs are not first-class inputs to validation and export.

### Completion Criteria

- Long-form generation produces canonical manifest rows.
- Manifest rows record segment count, segment frames, method, source start frame,
  final output path, and prompt.
- Long clips can be exported directly or split into labeled windows.
- Segment/window metadata is preserved for downstream training.

### Implementation Plan

1. Update `extend_video.py` to write canonical manifest rows for final outputs
   and optionally per-segment outputs.
2. Add SVI run manifest support, either through a wrapper script or a manifest
   registration command.
3. Add export options for `--long-mode whole` and `--long-mode windows`.
4. Implement window splitting metadata if detector training needs fixed-length
   clips.
5. Add tests for long-form manifest rows and export behavior.

## Issue 8: Category Support Is Inconsistent

`self_injury` exists in I2V generation but not in all class maps, preprocessing
choices, caption triggers, or export logic. Other categories have partial LoRA
paths but no complete data/config coverage.

### Completion Criteria

- Every enabled category has class id, prompt file, caption trigger, preprocessing
  support, generation support, export support, and validation support.
- Unsupported categories fail with a clear message.
- `self_injury` is either fully integrated or explicitly disabled.
- Category definitions are not duplicated across scripts.

### Implementation Plan

1. Create a central category config with class ids, triggers, prompt files, and
   supported generation methods.
2. Replace duplicated category lists in scripts with this config.
3. Add category validation at script startup.
4. Add tests that every enabled category has required config fields and files.
5. Decide whether `self_injury` is an enabled detector class and update export
   accordingly.

## Issue 9: Operational Configuration Is Scattered

Paths, ports, model names, LoRA names, GPU choices, dimensions, frame counts, and
workflow paths are duplicated across shell scripts, Python scripts, docs, and
workflow templates.

### Completion Criteria

- A single project config file defines default paths, models, workflows,
  categories, dimensions, frames, fps, and LoRA naming.
- Scripts can run from config without editing Python or shell source.
- Machine-specific paths are isolated and documented.
- A small smoke run can be launched from one command/config.

### Implementation Plan

1. Add a TOML or YAML project config, for example `configs/pipeline.toml`.
2. Add a config loader with environment-variable overrides for machine-specific
   paths and ports.
3. Migrate duplicated constants from generation, training, captioning, and export
   scripts into config.
4. Keep script flags as overrides over config defaults.
5. Add `PIPELINE.md` with the supported generate -> validate -> export workflow.
6. Add a smoke-test command that validates config and runs a non-GPU dry run where
   possible.
